Verbinde dein lokales Repository mit einem Remote-Repository und pushe Änderungen dorthin.

Remotes

Wenn du etwas auf GitHub legst, lebt diese Kopie auf einem von GitHub's Servern. Das macht es zu einem Remote-Repository, weil es nicht auf deinem Computer ist, sondern auf einem Server, "remote" und irgendwo anders. Durch Pushen deiner lokalen (auf deinem Computer) Änderungen dorthin, hältst du es auf dem neuesten Stand.

Andere können dann immer die neuesten Änderungen von deinem Projekt erhalten, indem sie deine Änderungen vom Remote herunterpullen (und auf ihren Computer). So kann jeder an einem Projekt zusammenarbeiten, ohne Zugang zu deinem Computer zu benötigen, wo deine lokale Kopie gespeichert ist.

Ein Remote-Repository erstellen

Du möchtest deine lokale Version mit einer synchronisieren, die auf GitHub.com gespeichert ist. Erstelle zuerst ein neues Remote Repository auf GitHub.com.

  • Gehe zu github.com, melde dich an und klicke auf das '+' oben rechts und dann auf 'New repository'.
  • Gib ihm einen Namen, der dem Namen deines lokalen Repositories entspricht, 'hello-world', und eine kurze Beschreibung.
  • Mache es öffentlich. Das bedeutet, es wird in deinem öffentlichen Profil aufgelistet.
  • Initialisiere nicht mit einem README, weil wir bereits eine Datei lokal namens 'readme.txt' haben. Das ist eine Hilfs- Option von GitHub, falls du es noch nicht gemacht hättest.
  • Lasse '.gitignore' und 'license' auf 'none' stehen. Wir werden sie in diesem Tutorial nicht verwenden.
  • Klicke auf 'Create repository'!

Readmes, .gitignores und Lizenzen

Das sind häufige Dateien in Open-Source-Projekten, daher bietet GitHub eine Option, sie für dich zu erstellen, um das Starten eines neuen Projekts zu erleichtern.

Eine Readme erklärt, was das Projekt ist, wie man es verwendet, und oft, wie man beitragen kann (obwohl es manchmal eine zusätzliche Datei 'CONTRIBUTING.md' für diese Details gibt).

Eine .gitignore ist eine Liste von Dateien, die Git nicht verfolgen sollte, zum Beispiel Dateien mit Passwörtern!

Eine Lizenz-Datei ist der Typ der Lizenz, den du auf dein Projekt legst. Das lässt andere wissen, wie sie es verwenden können. Informationen zu den Typen findest du hier: choosealicense.com.

Verbinde dein Lokales mit deinem Remote

Jetzt hast du ein leeres Repository auf GitHub.com gestartet. Oben siehst du 'Quick Setup', stelle sicher, dass der 'HTTPS' Button ausgewählt ist und kopiere die Adresse—das ist die Adresse deines Repositories auf GitHub's Servern.

Zurück in deinem Terminal und innerhalb des 'hello-world' Ordners, den du als Git-Repository in der vorherigen Challenge initialisiert hast, möchtest du Git sagen, sich die Adresse der Remote-Version auf GitHub's Servern zu merken. Du kannst mehrere Remotes haben, daher benötigt jedes einen Namen. Der primäre Remote wird typischerweise origin genannt.

Um einen Remote namens 'origin' zu deinem Repository hinzuzufügen:

git remote add origin <URLFROMGITHUB>

Dein lokales Repository weiß jetzt, wo dein Remote-Repository namens 'origin' auf GitHub's Servern lebt. Denke daran wie das Hinzufügen eines Namens und einer Adresse zur Schnellwahl—jetzt kannst du etwas dorthin senden, wenn du es brauchst.

Windows-Benutzer:

Falls du GitHub Desktop auf deinem Computer hast, wird ein Remote namens 'origin' automatisch in deinem lokalen Repository erstellt. In diesem Fall musst du ihm nur sagen, welche URL mit 'origin' verknüpft werden soll. Verwende diesen Befehl anstatt des 'add' Befehls oben:

git remote set-url origin <URLFROMGITHUB>

Arbeit zu deinem Remote pushen

Als nächstes möchtest du alles, was du lokal gemacht hast, zu deinem Remote-Repository auf GitHub pushen (senden). Das ist etwas, was du oft machen wirst, damit deine Remote-Version auf dem neuesten Stand ist und dem Zustand deiner lokalen Version entspricht.

Git hat ein Branching-System, damit du an verschiedenen Teilen eines Projekts zu verschiedenen Zeiten arbeiten kannst. Wir werden später mehr darüber lernen, aber standardmäßig heißt der erste Branch 'main'. Wenn du von einem Projekt pushst (und später pullst), sagst du Git den Branch-Namen, den du willst, und den Namen des Remotes, auf dem es lebt.

In diesem Fall senden wir unseren Branch namens 'main' zu unserem Remote auf GitHub namens 'origin'.

git push origin main

Gehe jetzt zur Seite deines Remote-Repositories auf GitHub.com und aktualisiere die Seite. Wow! Alles ist jetzt lokal genauso wie remote. Glückwunsch zu deinem ersten öffentlichen Repository!
